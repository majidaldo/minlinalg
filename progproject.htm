Introduction

Several modules were developed to demonstrate aspects of numerical programming. The C language (following C99 standard) was selected to execute the following tasks:

·        Matrix Market file reader

·        A module/framework for matrix operations

·        Linear system solvers

·        Solution of the diffusion equation

What follows is a description of each.

Matrix Market File Reader

An effort has been made to be able to robustly read any file the follows the Matrix Market specification. The specification allows for any combination of the following that describe the matrix:

-        Object: matrix or vector

-        Format: coordinate or array

-        Field: real, double, complex, integer, or pattern

-        Symmetry: general, symmetric, skew-symmetric, or hermitian

The challenge of programming for any combination (case) of the above in C is the need to specify the proper data structure in the code for reading and storing the data for each case. Each case is reflected in the format of the data line. The data line can contain up to two indexes for matrix data and up to two values for complex numbers.

The solution to this problem is to programmatically generate code that processes each case[1]. In the code, each case is specified by the tuple (object,format,field) where each entry is represented by an integer (The symmetry of the matrix does not affect the format).

Then to offer a generic interface to the stored data, the data are stored in the corresponding array i1, i2, and v (complex data is stored as a complex type). So when a variable is not needed the pointer to the array is NULL.

The feature of the code that allows a common interface to the data utilizes function pointers. The specific reading function can be retrieved at run-time by indexing with the tuple.

In addition, the reader incorporates checking the metadata for errors for robust file reading.

This generates readingfuncs.h, readingfuncs.c, and readingfuncptrs.h

2.      Use functions found in readmmx.c to read matrix market data:

a.      Put data into memory with readmmfile(pointer_to_file)

b.  Access the address of an element with getp2valueat(row,column,matrixmarketdata)

d.      Check content by printing an element with printmmvalat(pointer2value, fieldnum)

Matrix Operations Module

The matrix operations module contains the infrastructure for implementing matrix operations. A matrix’s data is held by a structure with the following members:

·        Data: an array holding data according to a matrix representation.

·        Matrix representation indicates how data is stored: row major, column major, constant (others can be added)

·        Matrix data type: double, float, complex, integer, and boolean

·        Size: number of rows and number of columns

·        Another member can be added to represent the matrix mathematical type such as sparse, dense, diagonal, constant, upper triangle…etc. (These can roughly correspond to the matrix representation) This information can help in optimizing procedures for matrix operations.

An element of the matrix is pointed to by an indexing function. An indexing function is implemented for each matrix representation.

At the lowest level are array operations which in turn are described as pair-wise element operations of specific mappings of the form sourceàdestination applying an operation f(source,destination). One-to-one and transpose mappings have been implemented. Operations implemented are multiplication, addition, and setting the destination equal to the source. However, these operations are only defined for doubles. Nonetheless, the infrastructure is in place to allow for type conversions.

At a higher lever are the mathematical operations on matrices which can be described by array operations (a matrix operation can be defined correctly by more than one set of array operations). Matrix multiplication for doubles is implemented as matrixmatrixop.

Moreover, some utility functions were developed for creating a new matrix createmeatrix, copying a matrix copymatrix, freeing data reserved by a malloc freematrixdata, and printing a matrix printmat.

Linear System Solvers

These solvers depend on the matrix operations module. The implemented solvers are Gaussian elimination, Jacobi, Gauss-Seidel, successive over relaxation, and conjugate gradient. Where applicable, the solvers are hardcoded with a tolerance, TOL, and a maximum number of iterations, MAXITER.

Solutions have been verified from resources on the world wide web.

Demo program: gcc solvers.c solversdemo.c matops.c

Concluding Remarks

Most features in C have been used to execute this work including pointers, structures, enums, asserts, preprocessor directives, and function pointers. However, more const declarations would have made the code safer where applicable. Furthermore, I found that an object-oriented approach would have been better suited to write the matrix operations module in order to better organize the code.

An effort has been made to make the code as modular as possible (isolation of functionality and many small functions). New functionality can be easily integrated without changing much of the original code.

C is capable of efficiently handling numerical operations. But the operations must be carefully coded to avoid memory leaks and segmentation faults. Using the gdb debugger was very helpful in finding these problems.
